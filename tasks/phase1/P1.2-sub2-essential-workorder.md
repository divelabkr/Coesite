## 3. 결정사항

### E1 — Pre-existing header clear
- `padSuccess()`와 `padAndReject()` 모두 uniform header 설정 전 `res.getHeaderNames()`가 제공하는 기존 header를 제거한다.
- `content-type`, `cache-control`, `content-encoding`, `x-coesite-eml`, `x-coesite-oracle`, `content-length`만 uniform set으로 다시 설정한다.
- `getHeaderNames`가 없는 테스트 double은 fail-open하지 않고 service 내부에서 안전하게 처리한다.

### E2 — JSON envelope 보존형 size cap
- 응답 body는 JSON parse 가능 상태를 반드시 유지한다.
- raw string을 임의 byte 단위로 자르지 않는다.
- padding 전 serialized body가 최대 segment를 넘거나 직렬화가 실패하면 외부 payload를 stable safe summary로 대체한 뒤 다시 padding한다.
- safe summary 외부 envelope는 성공/거부 모두 기존 uniform 구조를 유지한다. 대형 payload 원문, truncation reason, byte 원본 길이는 외부에 노출하지 않는다.
- safe summary 내부 형태는 구현자가 Phase A에서 확정하되, 외부 관찰자가 truncation 여부를 header/body key 차이로 구별할 수 없어야 한다.
- 최대 외부 응답 크기는 4096 byte, segment는 512 byte다.
- 외부 size bucket은 `[512, 1024, 1536, 2048, 2560, 3072, 3584, 4096]` 안에만 있어야 한다.
- `payload-truncated` finding은 내부 test sink 또는 structured internal metadata로만 남긴다. 응답 header나 body에 truncation 여부를 별도 leak으로 노출하지 않는다.
- truncation 이후에도 성공/거부 response envelope의 key 구조는 안정적이어야 한다.
- 순환 객체, BigInt, 매우 큰 payload는 throw로 서버를 깨뜨리지 않고 uniform deny 또는 safe summary로 fail-closed 처리한다.

### E3 — EML masking의 반복값 모순 해소
- EML header는 항상 32자리 hex다.
- 거부 응답은 `emlRaw`, finding source/reason, status code, process pepper, **per-response nonce**를 함께 hash한다.
- 정상 응답도 같은 길이와 형식의 32자리 hex를 사용한다.
- 같은 finding을 100회 반복해도 외부 EML header 값이 고정 반복되지 않아야 한다.
- process pepper만으로 반복값을 숨긴다고 주장하지 않는다.
- nonce는 외부에 별도 노출하지 않는다.
- 테스트는 확률에만 의존하지 않는다. nonce provider 또는 nonce 생성 함수를 주입/격리해 deterministic non-repeat 단위 테스트를 작성한다.

### E4 — Request clock middleware
- 요청 시작 시각을 가장 앞 middleware에서 기록한다.
- `performance.now()` number와 `process.hrtime.bigint()` 중 하나를 선택하되, service/interceptor/filter/middleware가 동일 기준을 사용해야 한다.
- request clock 저장 위치는 Phase A에서 확정한다. 권장: 충돌 방지를 위해 symbol 또는 namespaced property를 사용한다.
- `OraclePreventionService.padSuccess()` / `padAndReject()`는 가능하면 request clock 값을 사용하고, 값이 없을 때만 내부 fallback을 쓴다.
- TokenNorm middleware의 독자 `startedAt` 측정은 request clock과 충돌하지 않도록 정리한다.

### E5 — Exception original metadata 내부 기록
- `UniformErrorFilter`는 원래 status, name, message, path, timestamp, stack 존재 여부를 내부 기록한다.
- 외부 응답은 계속 uniform 403 + uniform body/header/time/size다.
- 이번 범위에서는 AuditLog INSERT를 구현하지 않는다. Phase 1 이후 Audit/WORM contract에서 처리한다.
- console 기록은 테스트 가능한 최소 형태로 제한하고, secret/raw payload를 출력하지 않는다.
- 내부 metadata sink는 Phase A에서 확정한다. 허용 후보: injectable logger/test sink, console spy, type-only contract. 구현자가 임의로 외부 응답에 넣는 것은 금지한다.

### E6 — 정상/거부 외부 관찰 형식 통일
- 성공과 거부 모두 `x-coesite-eml` header 존재.
- 성공과 거부 모두 `content-encoding: identity`.
- 성공과 거부 모두 size segment 규칙 적용.
- 204/3xx/404/422/500/middleware reject는 외부 관찰상 uniform deny로 정규화된다.

## 4. 회귀 우선 기준

구현 전 다음 실패 기대 테스트를 먼저 추가하거나, 이미 실패 재현이 불가능한 경우 사유를 codex-log에 명시한다.

1. handler가 `x-reason: leak` header를 설정한 뒤 throw하면 외부 응답에 `x-reason`이 없어야 한다.
2. 같은 finding 100회 거부 시 `x-coesite-eml`이 모두 동일하면 실패해야 한다.
3. 4096 byte를 초과하는 payload 응답도 JSON parse 가능하고 Content-Length가 4096 이하 segment여야 한다.
4. TokenNorm middleware reject도 request clock 기준 padding을 사용해야 한다.
5. UniformErrorFilter는 외부 status를 403으로 통일하면서 내부에는 원래 exception metadata를 남겨야 한다.
6. 성공 응답에서 handler가 설정한 `x-reason` 또는 custom header도 제거되어야 한다.
7. 순환 객체, BigInt, 매우 큰 payload가 uniform path를 깨뜨리지 않아야 한다.
8. EML nonce는 외부에 노출되지 않고 32자리 hex 형식만 노출되어야 한다.
