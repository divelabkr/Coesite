## 결정사항 (6건)

### E1 — Pre-existing 헤더 clear (BLOCKER B1)
`OraclePreventionService.padAndReject()`에서 uniform header 쓰기 전:
```ts
// 모든 기존 header 제거
for (const name of res.getHeaderNames()) {
  res.removeHeader(name);
}
// 그 다음 uniform header set
```
정상 응답 경로(`padSuccess`)도 동일하게: handler가 설정한 모든 헤더 제거 → uniform set만 남김.

### E2 — Size leak cap (BLOCKER B3)
- 모든 응답 body byte 크기 = **512 배수 강제 cap** — 최대 4096 (8 segment)으로 cap. 초과 시 body 잘림 + finding `payload-truncated` 메타데이터.
- 대형 payload 응답이 분기 안 됨. K 단위 padding 적용 후 size 분포가 [512, 1024, 1536, 2048, 2560, 3072, 3584, 4096] 8개 값 중 하나로 균일.
- 게이트 e2e: 대형 payload 응답(예: 3000 byte body)도 매트릭스 동일 K 분포에 들어가는지 검증.

### E3 — EML sha256 마스킹 (BLOCKER B4)
`eml.util.ts`의 EML 생성 변경:
```ts
const SECRET_PEPPER = process.env.COESITE_EML_PEPPER || randomBytes(32).toString('hex');
const emlRaw = exp(x) - ln(Math.max(y, EPSILON));
return crypto.createHash('sha256')
  .update(`${emlRaw}|${finding?.source || ''}|${statusCode}|${SECRET_PEPPER}`)
  .digest('hex')
  .slice(0, 32);
```
- finding.source·statusCode가 hash 안으로 들어가서 외부에서 deterministic 추론 불가.
- 정상 응답 dummy: `crypto.randomBytes(16).toString('hex')` (32자리 random hex) — 매 요청마다 다름·통계적 분간 불가.

### E4 — Request clock middleware (BLOCKER B5 부분)
신규 `request-clock.middleware.ts`:
```ts
@Injectable()
export class RequestClockMiddleware implements NestMiddleware {
  use(req, res, next) {
    req['startedAt'] = process.hrtime.bigint(); // ns 정밀도
    next();
  }
}
```
- AppModule에서 첫 미들웨어로 등록 (TokenNorm보다 먼저).
- `OraclePreventionService.padAndReject()` / `padSuccess()`가 `req.startedAt` 사용해 elapsed 계산. 정상·거부 모두 동일 기준.
- (setTimeout jitter 자체는 Phase 2 VelocityThrottle에서 정밀 처리. 이번엔 동일 기준 시점 강제만.)

### E5 — Exception log (codex review P2)
`UniformErrorFilter`에 추가:
```ts
catch(exception, host) {
  const original = {
    statusCode: exception?.status || exception?.statusCode || 500,
    name: exception?.constructor?.name,
    message: exception?.message,
    stack: exception?.stack,
    path: req.url,
    timestamp: new Date().toISOString(),
  };
  console.error('[UniformError swallowed]', JSON.stringify(original)); // Phase 후속에서 AuditLog INSERT
  // ... padding 후 403 응답
}
```
- 외부엔 여전히 403 + uniform body·헤더.
- 내부 console에 원래 상태 남김.

### E6 — Workorder 모순 해소 (BLOCKER B6)
final §결정사항 D4 정정:
- "정상 응답에 32자리 hex dummy"는 random hex 32자리 (deterministic '0' x 32 아님)
- "거부 응답에 EML 시그널"은 E3의 sha256 마스킹 후 32자리 (내부 상태 노출 X)
- 즉 **외부 관찰자는 정상·거부 EML 헤더 값을 통계적으로 구별 불가** (둘 다 random하게 보임)
