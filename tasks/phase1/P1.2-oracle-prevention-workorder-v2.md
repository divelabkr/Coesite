## 결정사항 — Pre-Phase A 14건 반영

### Codex BLOCKER

**B-C1 — 경로 정합** (Codex BLOCKER 1)
- `packages/api/src/common/oracle-prevention/` 사양 우선.

**B-C2 / B-G1 — Middleware exception padding 보장** (Codex BLOCKER 2 + Gemini BLOCKER 1)
- Padding 로직을 `OraclePreventionService.padAndReject()` 단일 함수로 통합.
- TokenNormMiddleware: `res.end()` 제거 → `await service.padAndReject(res, finding)` 호출 (middleware 안에서 직접 padding).
- UniformErrorFilter: 일반 exception은 동일 service의 동일 함수 호출.
- Interceptor: 성공 응답에도 동일 padding 적용.
- 즉 **세 경로 모두 같은 service 호출** → 응답 분포 균일.

**B-C3 — P8 모든 응답 범위** (Codex BLOCKER 3)
- 사양 P8 그대로: 200·201·403·404·422 (validation)·500·middleware exception·unmatched route 모두 균일.
- E2E 테스트로 각 상태 코드별 응답 시간·크기 통계 검증.

### Codex MAJOR

**M-C1 — EML 시그널 모든 거부 응답 첨부** (Codex MAJOR 1)
- `X-Coesite-EML` HTTP Header에 시그널 (Body 균일성 유지).
- 헤더는 거부 응답에만(403), 정상 응답엔 빈 헤더(같은 크기·구조).

**M-C2 — 통계 기준 사양 명시** (Codex MAJOR 2)
- E2E 통계: **1000회 요청·응답 시간 stddev < 5ms·크기 K(=512) 배수·t-test p > 0.05**.
- 게이트에 명시.

**M-C3 — 크기/형태 leak 게이트** (Codex MAJOR 3)
- `Content-Length` 동일·gzip 후 크기 동일·헤더 set 동일·error body 동일 (UniformError 메시지 단일화).
- 매트릭스 테스트 필수.

**M-C4 — AppModule runtime 테스트** (Codex MAJOR 4)
- grep 대신 e2e supertest: `request(app).get('/').expect(200)` 후 응답 header에 `X-Coesite-EML` 빈 값·padding 적용·body 정확 검증.

**M-C5 — Fixed-point DoS 상한** (Codex MAJOR 5)
- TokenNorm fixed-point: 최대 5회·각 회 입력 크기 추적·base64 디코드 결과가 원본 256배 초과 시 fail-closed·재귀 depth 3 초과 fail-closed.
- nested body·query·header 전체 적용 명시.

**M-C6 — E2E/load-test 영향 맵** (Codex MAJOR 6)
- `packages/api/test/e2e/oracle-prevention.e2e.test.ts` 신규 (위 영향 맵).

**M-C7 — NOT_RUN 회피** (Codex MAJOR 7)
- Docker 무관 sanity(`pnpm start` HTTP 응답 통계 e2e)는 **NOT_RUN 절대 불가**. Docker 의존 검증만 NOT_RUN 허용.
