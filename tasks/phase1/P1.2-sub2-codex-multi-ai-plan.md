## 2. 잔존 3건 처리 기준

### R1. E3 EML 모순
- 문제: 고정 pepper만 쓰면 같은 finding 반복 시 EML 값이 같아질 수 있다.
- 보정 기준: nonce 또는 요청 단위 입력을 hash에 포함해 같은 finding 반복도 외부 관찰값이 고정되지 않게 한다.
- 검증: 같은 finding 100회 반복 시 EML 값 다양성 확인. 정상·거부 헤더가 형식상 구별되지 않아야 한다.

### R2. 영향 맵 누락
- 문제: 기존 workorder가 수정 필요 파일을 모두 포함하지 않았다.
- 보정 기준: `types.ts`, `size-padding.util.ts`, `oracle-prevention.interceptor.ts` 등 실제 영향 파일을 영향 맵에 명시한다.
- 검증: Phase A에서 git/호출부/rules/문서/테스트 5축 재대조.

### R3. E2 size cap 사양 부족
- 문제: body truncation 방식과 JSON 무결성, 외부 size bucket 노출 기준이 모호하다.
- 보정 기준: JSON envelope 내부에서 payload를 cap 처리하고, 응답 전체는 512 byte segment, 최대 4096 byte로 제한한다.
- 검증: 4096 byte 초과 응답이 4096 byte 이하로 고정되고 JSON parse가 깨지지 않아야 한다.
