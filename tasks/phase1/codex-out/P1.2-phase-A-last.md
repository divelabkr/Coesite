## [P1.2 final] Codex 이해 확인 (Phase A)

1. 03-PROMPTS.md P1.2 사양 핵심 복창

P1.2의 목표는 `packages/api/src/common/oracle-prevention/`에 OraclePrevention 모듈을 작성해 P8 Oracle Prevention을 강제하는 것이다. 핵심은 최소 응답 시간 padding, K=512 단위 응답 크기 padding, 모든 거부 응답의 동일 메시지·동일 외부 status `403`, 성공/실패 응답 시간 분포 균일화다. 모든 거부 응답에는 `eml(x,y) = exp(x) - ln(y)` 기반 EML 시그널을 붙이며, 통계 테스트로 1000회 요청 기준 응답 시간 표준편차 `<5ms`, 응답 크기 K 배수를 검증한다. 제약은 P1·P2 준수, P8 핵심 구현체이며, 검증은 vitest 통계 테스트와 로컬 부하 테스트다.

5. Docker 부재 시 처리

Docker 부재를 이유로 6·7을 NOT_RUN 처리하지 않는다. NestJS in-process e2e와 supertest 기반 검증은 Docker 없이 PASS 필수로 수행한다. Docker 의존 검증이 실제로 필요한 항목만 명시적으로 분리하고, 미실행 사유를 검증 결과에 그대로 남긴다.

6. **작업지시 vs 사양 정합 의문점** (필수)

- EML `number → 32자리 hex` 변환 규칙이 구체화되어 있지 않다. 정수 스케일링, 해시화, 버퍼 인코딩 중 어떤 방식인지 결정 필요.
- `Content-Length 동일`과 `body byte 크기 512 배수`는 같은 조건이 아니다. 정상·거부 모두 동일 segment 개수까지 맞출지 기준이 필요하다.
- 성공 응답을 `200/201`로 한정했는데 기존 API의 `204`, redirect, stream/buffer 응답이 있다면 처리 정책이 필요하다.
- `\0` padding은 JSON 응답 소비자에게 영향을 줄 수 있다. body 포맷을 유지하는 padding 방식인지, raw body padding인지 확인 필요.

8. **결정사항 모순 후보** (필수)

- D3의 “정상 응답 200/201”과 “모든 응답 body padding”은 204/no-content 응답 정책에서 충돌 가능성이 있다.
- D6의 “K 배수”와 게이트 7의 “Content-Length 동일”은 강도가 다르므로 구현 기준을 하나로 고정해야 한다.
- D7의 최소 200ms와 게이트 7의 `stddev<5ms`는 로컬 부하 환경에서 이벤트 루프 지연에 취약할 수 있다.
- UniformErrorFilter가 모든 404/422/500을 403으로 바꾸더라도 middleware 단계에서 직접 종료하는 경로와 완전히 같은 header/body가 되는지 별도 보장이 필요하다.

9. 위험 3개

- 응답 크기 padding이 API body 형식을 깨뜨릴 위험.
- 시간 분포 통계 테스트가 환경 노이즈로 flaky해질 위험.
- TokenNorm fixed-point 확장 중 DoS 상한·metadata 누출을 놓칠 위험.

10. 레드팀 우선 시각

1순위는 size leak이다. `Content-Length`, header byte length, compression, padding 방식 차이가 바로 관찰 가능하다.  
2순위는 timing leak이다. middleware/filter/interceptor 경로별 처리 시간이 갈라질 수 있다.  
3순위는 form leak이다. exception message, stack, validation shape, unmatched route shape가 외부에 남으면 P8이 깨진다.
