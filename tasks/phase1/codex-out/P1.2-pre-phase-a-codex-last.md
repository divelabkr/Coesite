**BLOCKER**

1. **OraclePrevention 위치 충돌**
   `docs/03-PROMPTS.md` P1.2 목표는 `packages/api/src/common/oracle-prevention/`인데, 작업지시는 `packages/api/src/meta-layer/oracle-prevention/`로 지정합니다. `docs/01-CLAUDE.md` 디렉터리 구조도 OraclePrevention을 `common` 횡단 관심사로 둡니다. D2가 “사양 우선”이라고 하면서 영향 맵이 사양과 충돌합니다.

2. **TokenNorm rejection이 OraclePrevention padding을 탄다는 가정이 불완전**
   작업지시는 `TokenNormMiddleware`에서 `ForbiddenException`을 throw하면 `UniformErrorFilter`와 `OraclePreventionInterceptor` padding을 거친다고 봅니다. 하지만 NestJS에서 middleware는 interceptor 앞단입니다. 이 경로가 실제로 interceptor post-processing을 통과한다는 보장이 작업지시에 없습니다. P1.1 B4 흡수가 false PASS 될 수 있습니다.

3. **P8 “모든 응답” 범위가 게이트에서 빠짐**
   헌법 P8은 모든 응답의 시간·크기·형태 균일화입니다. 작업지시 게이트는 정상 200과 악의 403 중심이고, 404, 500, validation error, middleware exception, Nest 내부 예외, unmatched route를 필수 수용 기준으로 못 박지 않습니다. P8 핵심 누락입니다.

**MAJOR**

1. **EML 요구 약화**
   P1.2 사양은 “모든 거부 응답에 EML 시그널 첨부”를 요구하지만, 작업지시는 “핵심 분기에서 util 호출”로 낮아졌습니다. 응답에 붙이는지, 내부 메타데이터인지, `x/y` 비노출과 어떻게 양립하는지 불명확합니다.

2. **통계 게이트 기준 불명확**
   사양은 1000회 요청, 응답 시간 표준편차 `< 5ms`, 크기 K 배수를 명시합니다. 작업지시는 “통계적으로 구별 불가”, “5+ 케이스”로 표현해 임계값 없는 PASS를 허용합니다.

3. **크기·형태 leak 게이트 부족**
   `Content-Length`, gzip 후 크기, 헤더 차이, Nest 기본 error body 차이를 레드팀 관점에만 두고 필수 게이트로 올리지 않았습니다.

4. **AppModule 검증이 grep 기반**
   `grep TokenNormModule` 등은 주석·문자열만 있어도 통과합니다. 실제 DI 등록, middleware 적용, global interceptor/filter 동작을 런타임 테스트로 검증해야 합니다.

5. **fixed-point 흡수안의 DoS/상한 기준 부족**
   최대 5회 cap은 있으나 입력 크기, 재귀 depth, base64 팽창, nested body/query/header 전체 적용 기준이 없습니다. “cap 초과 fail-closed”만으로는 성능 leak·DoS 방어가 닫히지 않습니다.

6. **영향 맵에 E2E/load-test 파일 누락**
   P1.2 검증에는 실제 HTTP 요청 분포 테스트가 필요하지만 영향 맵은 unit test 중심입니다. test app/controller, e2e/supertest 또는 k6 스크립트 위치가 빠졌습니다.

7. **`Docker 부재 시 NOT_RUN` 문구가 런타임 검증 회피 가능**
   P1.2 핵심은 runtime oracle 검증입니다. Docker와 무관한 `pnpm start`/HTTP sanity까지 NOT_RUN처럼 처리될 여지가 있습니다.

**권고 보강 5건**

1. OraclePrevention 경로를 `packages/api/src/common/oracle-prevention/`로 맞추거나, 의도적 변경이면 사양 변경 근거를 명시.
2. TokenNorm 거부 경로가 실제 padding을 거치도록 middleware 자체 padding 또는 guard/interceptor 전환 중 하나로 고정.
3. 200/403/404/500/validation/malformed URL 응답 매트릭스를 필수 게이트로 추가.
4. 통계 기준을 사양 그대로 명시: 1000회, stddev `< 5ms`, body/header/Content-Length/K-segment 검증, compression 처리 기준 포함.
5. EML signal의 저장/응답 위치, `y > 0`, overflow/NaN fail-closed 기준을 명시.
