## [P1.1] Codex 이해 확인 (Phase A)

1. 03-PROMPTS.md P1.1 사양 핵심 복창 (한 줄씩)
- Phase 1 Task 1은 `TokenNorm 미들웨어`다.
- META-LAYER 첫 관문으로 모든 incoming 요청의 token을 정규화한다.
- 공격자의 인코딩 변형 우회를 차단한다.
- 목표 산출물은 `packages/api/src/meta-layer/token-norm/` 모듈이다.
- `TokenNormService`는 Unicode NFKC 정규화를 수행한다.
- `TokenNormService`는 제로폭 문자 `U+200B`, `U+200C`, `U+200D`, `U+FEFF`를 제거한다.
- `TokenNormService`는 Cyrillic→Latin 매핑 테이블로 동형이의 문자를 정규화한다.
- `TokenNormService`는 의심 패턴에 대해 Base64 디코딩을 시도한다.
- `TokenNormService`는 URL 디코딩 및 이중 인코딩 탐지를 수행한다.
- `TokenNormMiddleware`는 NestJS 미들웨어로 모든 라우트에 적용한다.
- `TokenNormMiddleware`는 `req.body`, `req.query`, `req.headers`를 정규화한다.
- 변형 탐지 시 metadata를 기록하며 P5 AttestationChain 연동을 고려한다.
- 단위 테스트는 20+ 케이스가 필요하다.
- 테스트에는 정상 케이스, 제로폭 문자 삽입, Cyrillic `а` vs Latin `a`, Base64 인코딩 페이로드, 이중 URL 인코딩이 포함된다.
- 제약 P1: ML 사용 금지, 룰 기반만 허용한다.
- 제약 P2: 변형 탐지 시 거부하며 fail-closed로 처리한다.
- 제약 P8: 정규화 처리 시간을 측정한 뒤 padding하여 응답 시간을 균일화한다.
- 검증은 vitest 20+ 통과, tsc 0 오류, scan 0 위반이다.
- 산출물 보고가 필요하다.

7. 위험 3개
- Unicode normalization, bidi, composed/decomposed 문자로 TokenNorm 우회 가능성.
- fail-closed 거부 응답의 시간·크기·형태 차이로 oracle leak 가능성.
- `req.headers` 정규화가 인증·프록시·콘텐츠 타입 처리와 충돌할 가능성.
