## 결정사항

### D1. P1.1 BLOCKER 4건 동시 처리 (Gemini 추천 아키텍처)
- **B1 AppModule 통합** — `app.module.ts`에 두 모듈 등록 + global interceptor·filter 등록
- **B2 Fixed-point 정규화** — `token-norm.service`의 `normalize()` 호출을 변화 없을 때까지 루프 (최대 5회 cap, 초과 시 fail-closed)
- **B3 Malformed URL fail-closed** — `decodeURIComponent` throw 시 finding으로 분류·rejection
- **B4 대칭 padding** — TokenNorm middleware의 `res.end()` 제거 → `throw new ForbiddenException(...)` → uniform-error.filter가 잡아 OraclePreventionInterceptor의 padding 거침

### D2. 03-PROMPTS.md P1.2 사양 우선
- Codex는 `docs/03-PROMPTS.md` P1.2 섹션을 정확히 읽고 모든 요구사항 적용. 본 작업지시는 그 위에 영향 맵·결정·게이트·금지·레드팀 추가.

### D3. P8 균일화 (헌법 본질)
- **시간**: 모든 응답 최소 시간 padding (예: 200ms·configurable). 정상·거부 동일.
- **크기**: K-segment 단위 응답 크기 padding (예: 512바이트 배수).
- **형태**: UniformError로 모든 거부 = 동일 메시지·동일 status code (403).
- **EML 트리**: `eml(x,y) = exp(x) - ln(y)` util로 핵심 분기에서 통일 호출.

### D4. 테스트 기준 (사양 + 강화)
- 03-PROMPTS.md P1.2 사양 명시 수. **+** 통계 분포 테스트(정상·거부 응답 시간·크기 분포가 통계적으로 구별 안 되는지) 필수 5+ 케이스.
- TokenNorm 회귀 — P1.1 29개 + fixed-point/malformed URL 추가.

### D5. NestJS 통합
- `OraclePreventionInterceptor`는 `APP_INTERCEPTOR` provider로 글로벌 등록.
- `UniformErrorFilter`는 `APP_FILTER` provider로 글로벌 등록.
- TokenNormMiddleware는 `MiddlewareConsumer`로 모든 라우트.
- 실행 순서: middleware (TokenNorm) → guard → interceptor (Oracle padding) → controller → interceptor (post) → filter (UniformError if exception).

## 레드팀 관점 (Phase C — 10개)
1. **P8 timing leak** — CPU usage·이벤트 루프 지연으로 padding 우회
2. **P8 size leak** — Content-Length·압축(gzip) 후 크기 차이·헤더 차이
3. **P8 form leak** — error code·exception 패턴 차이 (NestJS 내부 message)
4. **Fixed-point limit** — 5회 cap 초과 시 fail-closed 정확한가·DoS
5. **Malformed URL 변형** — `%2`·`%`·multi-byte UTF-8 깨짐 등 다른 변형
6. **Middleware exception → filter** — TokenNorm exception이 진짜 OraclePrevention padding 거치는지
7. **AppModule global 적용** — controller 없는 상태에서 진짜 작동·404와 정상 응답 구별
8. **EML 트리 실제 호출** — 통일 호출이라고 명시했지만 실제로 모든 분기에서 호출되는지
