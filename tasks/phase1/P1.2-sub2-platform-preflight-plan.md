## 2. Preflight Gate 순서

### Gate 0 — 작업 기준점 / 환경

1. `git status --short`
2. `command -v codex`
3. `command -v gemini`
4. `gemini --version`
5. Gemini auth 확인
6. `TMPDIR=/tmp` 강제 여부 확인

중단 조건:
- branch/HEAD/worktree가 workorder와 다르면 중단.
- dirty worktree가 workorder 영향 맵 밖이면 중단.
- Gemini 인증이 없으면 `Gemini NOT_RUN`으로 기록하고, 구현 GO 근거로 쓰지 않는다.
- `pnpm test`를 `TMPDIR=/tmp` 없이 실행하지 않는다.

### Gate 1 — 정적/빌드 게이트

1. `pnpm -r build`
2. `pnpm -r exec tsc --noEmit`
3. `bash scripts/security-wall.sh`
4. `bash scripts/scan-principles.sh`

중단 조건:
- 하나라도 FAIL이면 구현 금지.
- `scan-principles.sh` PASS를 P8/P5/P10 완전성 증명으로 해석하지 않는다.

### Gate 2 — 테스트 실행성 분리

1. `pnpm test`를 먼저 실행하지 않는다. 이 환경에서는 Windows temp 경로 문제로 수집 전 실패할 수 있다.
2. `TMPDIR=/tmp pnpm test`를 sandbox 내부에서 실행한다.
3. `listen EPERM`이면 sandbox 환경 실패로 기록한다.
4. 필요 시 승인된 sandbox 밖에서 `TMPDIR=/tmp pnpm test`를 재실행한다.
5. sandbox 밖 실행도 불가하면 사람 waiver 없이는 구현 NO-GO.

기록 형식:

```markdown
- `pnpm test`: FAIL/PASS/NOT_RUN — 사유
- `TMPDIR=/tmp pnpm test` sandbox: FAIL/PASS/NOT_RUN — 사유
- `TMPDIR=/tmp pnpm test` sandbox outside: FAIL/PASS/NOT_RUN — 사유
```

### Gate 3 — 회귀 테스트 선행

구현 전 반드시 실패 기대 테스트를 먼저 추가한다.

| # | 회귀 테스트 | 상태 |
|---|---|---|
| 1 | throw 후 handler/custom header 제거 | Pending |
| 2 | 같은 finding 100회 EML non-repeat | Pending |
| 3 | 4096B 초과 payload JSON parse + cap | Pending |
| 4 | TokenNorm reject request clock 공유 | Pending |
| 5 | UniformErrorFilter original metadata internal sink | Pending |
| 6 | 성공 응답 custom header 제거 | Pending |
| 7 | circular object / BigInt / huge payload fail-closed | Pending |
| 8 | nonce 외부 미노출 + 32 hex만 노출 | Pending |

중단 조건:
- 회귀 테스트 없이 구현부터 시작하면 NO-GO.
- 실패 재현이 불가능한 항목은 codex-log에 사유를 남긴 뒤 사람 승인 필요.

### Gate 4 — Coverage / Fast-Heavy 분리

현재 coverage 80% 기준은 문서에 있으나 package/CI에서 강제되지 않는다.

선택지:
1. A-1 작업으로 coverage script와 CI job을 먼저 추가한다.
2. 이번 P1.2-sub2에서는 coverage gate를 `NOT_RUN waiver`로 명시하고, 구현 GO 근거로 쓰지 않는다.

Fast gate:
- build
- tsc
- unit tests
- lightweight runtime e2e
- scan-principles
- security-wall

Heavy gate:
- timing distribution e2e
- 1000회 통계 테스트
- memory/DoS
- huge payload
- coverage threshold

### Gate 5 — RedTeam

레드팀 Codex는 Deep 시간 예산으로 실행한다. P1.2-sub2는 runtime oracle, 회귀 테스트, 사업 전 보안 판단과 연결되므로 quick 검증을 GO 근거로 쓰지 않는다.

시간 예산:

| 모드 | 예산 | 이번 작업 적용 |
|---|---:|---|
| runtime-redteam Deep | 45~60분 | 필수 |
| platform-redteam Deep | 45~60분 | dependency/CI/business preflight 동반 시 필수 |
| timeout | NOT_RUN/PARTIAL | GO 근거 사용 금지 |

레드팀 Codex는 아래 matrix를 최소 산출물로 낸다.

| 축 | 확인할 공격 |
|---|---|
| header | pre-existing/custom header leak, content-length mismatch |
| size | 512 bucket oracle, 4096 cap 초과, truncation oracle |
| time | success/reject/exception/middleware timing 차이 |
| exception | original status/name/message/path leak |
| metadata | internal finding/stack/raw payload 외부 노출 |
| memory/DoS | circular, BigInt, huge payload, stringify failure |
| scan bypass | P8/P5/P10 false positive/false negative |

중단 조건:
- RedTeam이 NOT_RUN이고 사람 waiver가 없으면 구현 NO-GO.
- P0/P1이 남으면 구현 NO-GO.
- Quick 또는 timeout 결과만 있으면 구현 GO 근거로 쓰지 않는다.

### Gate 6 — Gemini CLI

현재 확인:
- `gemini` binary: 존재
- `gemini --version`: `0.42.0`
- `HOME=/tmp gemini ...`: auth 설정 없음으로 실행 차단

중단 조건:
- Gemini NOT_RUN은 허용 가능하지만, 반드시 기록한다.
- Gemini 결과를 구현 GO 조건으로 쓰려면 auth 설정 후 재실행해야 한다.

## 4. 구현 GO 조건

구현 GO는 아래가 모두 만족될 때만 가능하다.

- Gate 0 환경 기준점 기록 완료.
- Gate 1 build/tsc/security-wall/scan PASS.
- Gate 2 테스트 실행성 분리 완료.
- Gate 3 회귀 테스트 선행 계획 승인.
- Gate 4 coverage 정책이 A-1 선행 또는 waiver로 확정.
- Gate 5 RedTeam 완료 또는 사람 waiver.
- Gate 6 Gemini 실행 또는 NOT_RUN 기록 + 사람 waiver.
- Phase A에서 safe summary envelope, request clock storage, metadata sink를 확정.
