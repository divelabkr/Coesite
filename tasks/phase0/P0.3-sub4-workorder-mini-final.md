## 결정사항

**M1 — temp table schema 공격 차단** (codex review·레드팀 공통 BLOCKER)
- `verify_chain_insert()` 함수의 동적 SQL을 schema-qualified로 변경:
  ```sql
  -- 변경 전: EXECUTE format('SELECT EXISTS(SELECT 1 FROM %I WHERE hash = $1)', TG_TABLE_NAME)
  -- 변경 후: EXECUTE format('SELECT EXISTS(SELECT 1 FROM %I.%I WHERE hash = $1)', TG_TABLE_SCHEMA, TG_TABLE_NAME)
  ```
- AuditLog scope 검증 동적 SQL도 동일하게 `TG_TABLE_SCHEMA + TG_TABLE_NAME` 패턴.
- GENESIS 중복 검증 동적 SQL도 동일.

**M2 — pnpm-lock 동기화** (codex review·레드팀 공통 BLOCKER)
- `pnpm install` 실행해 `dotenv-cli` 의존성 lock에 반영.
- 게이트: `pnpm install --frozen-lockfile` 통과 확인.

**M3 — Multi-agent scope 자연키** (Gemini BLOCKER)
- 4개 WORM 테이블의 `prevHash` global unique를 자연키 scope로 변경:
  - `WormLog`: `(source, prevHash)` UNIQUE — 기존 global prevHash unique 제거 후 복합 추가
  - `AdminActionLog`: `(adminId, prevHash)` UNIQUE
  - `DmsTriggerLog`: `(triggerType, prevHash)` UNIQUE
  - `ProofBundle`: `(sessionId, prevHash)` UNIQUE
- INSERT 트리거 `verify_chain_insert()`도 테이블별 scope-aware로 강화:
  - AuditLog: agentId scope (기존 유지)
  - WormLog: source scope
  - AdminActionLog: adminId scope
  - DmsTriggerLog: triggerType scope
  - ProofBundle: sessionId scope
- GENESIS 중복 검증도 각 scope별로 (예: WormLog는 같은 source 내 GENESIS 1개).
