## [P0.3] Codex 이해 확인 (Phase A)

1. 영향 맵 대조: ✅

2. D1 8개 모델 핵심 필드 복창  
- 비-WORM: `Agent`, `Session`, `Policy`
- WORM 5개: `AuditLog`, `WormLog`, `AdminActionLog`, `DmsTriggerLog`, `ProofBundle`
- WORM 모델은 `hash`, `prevHash`, `createdAt` 중심의 증거 체인 구조를 가진다.

3. D2 공통 규약 복창  
- 모든 ID는 `cuid()`
- 모든 WORM 모델의 `prevHash`는 `NOT NULL`
- genesis는 `"GENESIS"` 상수로 입력
- WORM 모델에는 `@@map` 및 `/// WORM: UPDATE/DELETE forbidden by trigger`
- 필요한 인덱스: `agentId`, `createdAt`, `hash`
- `createdAt @default(now())`, `Agent.updatedAt @updatedAt`

4. D3 WORM 트리거 SQL 패턴 복창  
- `prevent_update_or_delete()` 함수 생성
- `BEFORE UPDATE OR DELETE` 트리거 적용
- 대상: `AuditLog`, `WormLog`, `AdminActionLog`, `DmsTriggerLog`, `ProofBundle`
- 위반 시 `P0001` 예외 발생

5. D4·D5 datasource·scripts 복창  
- datasource: PostgreSQL, `DATABASE_URL`
- generator: `prisma-client-js`
- 루트 scripts:
  - `prisma:generate`
  - `prisma:migrate`
  - `prisma:reset`
- `packages/api/package.json`:
  - deps: `@prisma/client`
  - devDeps: `prisma`

6. 8게이트 명령어 복창  
- `pnpm install`
- `pnpm prisma:generate`
- `pnpm dlx prisma validate --schema=prisma/schema.prisma`
- `prisma format`
- `prisma migrate dev`
- WORM UPDATE/DELETE 차단 검증
- `tsc --noEmit -p packages/api`
- Mountain 키워드 0건 검사
- `pnpm -r build`
- `pnpm test`

7. Docker 부재 시 처리  
- 게이트 4, 5는 `NOT_RUN`
- 사유를 요약본에 정직하게 기록

없음

9. 예상 위험 3개  
- `prevHash` 무결성은 schema만으로 완전 보장되지 않을 수 있음
- WORM 트리거가 `TRUNCATE`나 owner 권한 우회까지 막지 못할 수 있음
- migration 순서가 어긋나면 트리거 생성 시 테이블 부재로 실패 가능
