## [P0.3 sub-3] Codex 이해 확인 (Phase A)

1. 영향 맵 대조
- 수정 허용: `prisma/migrations/20260522000002_chain_integrity/migration.sql`, `prisma/migrations/20260522000003_role_separation/migration.sql`, `package.json`, `packages/api/package.json`
- 주의: C2/C5에 README 언급이 있으나 영향 맵에는 README가 없고, 절대 금지에 “영향 맵 밖 파일 수정 금지”가 있음.

2. C1~C5 처리 방안 한 줄씩
- C1: `AuditLog`는 `agentId` scope로 chain 검증, 나머지 WORM 테이블은 `prevHash` 전역 UNIQUE로 fork 방지.
- C2: role 생성 migration에서 고정 password 제거, `PASSWORD NULL`로 비활성 생성.
- C3: `CREATE ROLE`을 `DO $$ ... IF NOT EXISTS ...`로 idempotent 처리.
- C4: 5개 WORM 테이블의 `hash` CHECK에서 `hash = 'GENESIS'` 허용 제거.
- C5: `dotenv-cli` 추가 후 Prisma scripts가 `packages/api/.env`를 로드하도록 변경.

3. 7게이트 명령어
- `pnpm exec dotenv -e packages/api/.env -- prisma validate --schema=prisma/schema.prisma`
- `pnpm exec dotenv -e packages/api/.env -- prisma format --schema=prisma/schema.prisma`
- `rg "agentId|prevHash|PASSWORD NULL|IF NOT EXISTS|GENESIS" prisma/migrations/20260522000002_chain_integrity/migration.sql prisma/migrations/20260522000003_role_separation/migration.sql`
- `pnpm exec dotenv -e packages/api/.env -- prisma migrate dev --schema=prisma/schema.prisma`
- Docker/DB 가능 시 INSERT trigger SQL 시나리오 실행, 불가 시 `NOT_RUN`
- `pnpm install --lockfile-only` 또는 설치 후 `pnpm why dotenv-cli`
- `pnpm -r build`; `pnpm test`; Mountain 0건 확인

4. 의문점
- README 수정 지시가 C2/C5에 있으나 영향 맵 밖 파일 수정 금지와 충돌함. 현재 지시 기준으로는 README 수정하지 않는 것이 맞음.

5. 위험 3개
- 트리거 함수 분기에서 `AuditLog`와 비-`agentId` 테이블 정책이 섞이면 fork 방지가 깨질 수 있음.
- `hash != GENESIS` 변경 시 기존 fixture/seed/test data가 `hash='GENESIS'`를 쓰면 migration 또는 테스트가 실패할 수 있음.
- Docker/DB 미가용이면 migrate 및 INSERT trigger 검증은 `NOT_RUN`으로 남아 실제 DB 동작 검증 공백이 생김.

6. 약속 4개 [x]
- [x] 구현 전 영향 맵 밖 파일은 수정하지 않는다.
- [x] 운영 layer 변경은 하지 않는다.
- [x] DB 검증은 생성 → 검증 → 승인 → 실행 구조를 유지한다.
- [x] 미검증/실패/건너뜀은 요약 시 그대로 명시한다.
