## 결정사항

**C1 — Chain fork 방지** (codex review P1-2 + 레드팀 B4)
- INSERT 트리거 `verify_chain_insert`를 agentId-aware로 강화:
  - 테이블에 `agentId` 컬럼이 있으면(`AuditLog`) → prevHash 검증은 `WHERE agentId = NEW.agentId` scope. GENESIS 중복 검증도 agentId scope.
  - 테이블에 `agentId` 없으면(`WormLog`, `AdminActionLog`, `DmsTriggerLog`, `ProofBundle`) → **prevHash globally UNIQUE** (테이블 전체). `(agentId, prevHash)` unique 제약 → 단일 `prevHash` unique로 변경.
- 트리거 함수에 `TG_TABLE_NAME` 기반 분기 또는 테이블별 트리거 함수 분리.

**C2 — Role password 분리** (codex review P1-1 + 레드팀 B1)
- `20260522000003_role_separation/migration.sql`의 `CREATE ROLE ... PASSWORD '...'` 라인에서 **고정 placeholder password 제거**:
  ```sql
  -- migration은 role 생성만, password 설정은 별도 절차
  DO $$ BEGIN
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname='coesite_runtime') THEN
      CREATE ROLE coesite_runtime LOGIN PASSWORD NULL;  -- NULL = 비활성. 운영 시 ALTER ROLE로 설정.
    END IF;
  END $$;
  ```
- README에 추가: "WORM runtime 활성화 절차 (Phase 1+)" 섹션 — `ALTER ROLE coesite_runtime PASSWORD :secret_from_kms;` 가이드 + `.env`에 `RUNTIME_DATABASE_URL` 채우는 단계.

**C3 — Idempotent CREATE ROLE** (codex review P2-3 + 레드팀 B3)
- 위 C2의 `DO $$ ... IF NOT EXISTS ...`로 자동 흡수.

**C4 — `hash != 'GENESIS'` CHECK** (codex review P2-1)
- chain_integrity migration의 hash CHECK 제약:
  ```sql
  -- 변경 전: CHECK (hash ~ '^[a-f0-9]{64}$' OR hash = 'GENESIS')
  -- 변경 후: CHECK (hash ~ '^[a-f0-9]{64}$')  -- GENESIS는 prevHash 전용 sentinel
  ```
- 5개 WORM 테이블 모두 동일 변경.

**C5 — Prisma scripts .env 로드** (codex review P2-2)
- `packages/api/package.json` devDeps에 `dotenv-cli` 추가.
- 루트 `package.json` scripts 변경:
  ```json
  "prisma:generate": "dotenv -e packages/api/.env -- prisma generate --schema=prisma/schema.prisma",
  "prisma:migrate": "dotenv -e packages/api/.env -- prisma migrate dev --schema=prisma/schema.prisma",
  "prisma:reset": "dotenv -e packages/api/.env -- prisma migrate reset --schema=prisma/schema.prisma --force"
  ```
- README의 quick start에 prisma 실행 명령 그대로 동작하도록 검증.
