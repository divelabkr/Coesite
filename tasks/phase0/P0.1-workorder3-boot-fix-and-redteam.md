## 4. Phase B 결정사항 (확정 — 그대로 따를 것)

### D1. 의존성 추가
- `packages/api/package.json` dependencies에 `@nestjs/platform-express` 추가.
- 버전: 다른 NestJS 패키지와 정합되는 `10.4.15` (또는 NestJS 10.4.x 호환 최신).
- `pnpm install` 재실행 → `pnpm-lock.yaml` 갱신.

### D2. NestJS skel 보강 (최소 변경)
- `packages/api/src/main.ts`: 현재 `NestFactory.create(AppModule)` 호출은 유지하되, **명시적으로 Express 플랫폼 사용을 표현해도 좋고 안 해도 좋음**. 기본 어댑터 자동 사용 가능하므로 코드 변경 최소화 권장.
- `app.module.ts`: 변경 없음 (빈 모듈 유지).
- Controller / Service / DTO / Pipe 추가 금지 (이번 범위 밖).

### D3. 게이트 보강 — 부팅 sanity 추가
- 기존 5개 게이트 + 1개 추가 = 6개.
- 부팅 sanity 게이트:
  - 명령: `pnpm --filter @coesite/api start` 를 백그라운드로 띄움.
  - 3~5초 대기 후 프로세스 상태 확인.
  - 종료 코드 0이 아닌 즉시 실패 (`No driver (HTTP)` 류 stderr 검출 시 명확히 실패 보고).
  - 정상이면 프로세스 종료 (`Stop-Process` 또는 SIGTERM).
- 이 게이트는 통합 요약본 게이트 결과 표에 명시.

## 6. Phase C — 레드팀 관점 (Codex가 적대적으로)

다음 관점들로 **현재 P0.1 코드·구성·요약본을 깨뜨릴 시나리오** 발굴:

1. **공급망(Supply chain) 시각** — `pnpm-lock.yaml` 의 transitive deps 중 GPL/AGPL/SSPL 또는 의심 라이선스 있는지. 알려진 CVE가 있는 버전 핀이 있는지.
2. **빌드·런타임 분기 시각** — `pnpm install --frozen-lockfile` 으로 재현 가능한지. 로컬 캐시·`.pnpm-store` 없이도 통과하는지.
3. **F12 시크릿** — `.env`·`.env.example`·하드코딩된 토큰·키 패턴 있는지. `.gitignore`가 실제로 그것들을 가리는지.
4. **P6 결합의존 우회** — 누군가 `@coesite/api`를 `@coesite/types` 또는 `@coesite/utils` 없이 단독으로 빌드 시도하면 어떻게 막히는가. 단방향 의존(`api → types/utils`) 이 깨질 여지 있는지.
5. **P7 환각·미사용 path** — `tsconfig` 안에 실제 디렉토리와 불일치하는 `include`·`exclude`·`paths` 있는지.
6. **Wall 우회** — Security Wall Tier 1 변형 키워드가 deps 이름·라이선스 파일·README에 우연히라도 들어왔는지 (예: 패키지 이름 충돌).
7. **NestJS skel의 운영적 결함** — `main.ts`의 포트가 하드코딩이면 충돌 가능성, listen 실패 시 에러 처리 누락 등.
8. **다음 Phase가 깨질 시드** — P0.2 docker-compose, P0.3 Prisma, P0.4 CLAUDE.md 복사 시 이번 골격에서 충돌 가능성 (예: tsconfig·script 이름 충돌).

발견 항목은 **심각도(BLOCKER / MAJOR / MINOR / SUGGESTION)** 로 분류해 요약본에 작성.

**중요**: 레드팀은 "찾기"만. 실제 코드 수정은 별도 작업지시로 처리 (이번 사이클에선 BLOCKER 발견 시 통합 요약본에 명시하고 Claude 가이드 단계로 넘김).
