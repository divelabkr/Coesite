### D2. 공통 규약
- 모든 ID: `cuid()` (Prisma 권장).
- 모든 WORM 모델: `prevHash` **NOT NULL** (genesis 블록만 예외: 별도 상수 `"GENESIS"`로 입력).
- 모든 WORM 모델에 `@@map` + Prisma 주석 `/// WORM: UPDATE/DELETE forbidden by trigger`.
- 인덱스: `@@index([agentId])`, `@@index([createdAt])`, `@@index([hash])` (해당 필드 있는 모델).
- timestamp: `createdAt DateTime @default(now())`, `updatedAt DateTime @updatedAt` (Agent만).

### D3. WORM 트리거 (`prisma/migrations/V_002_worm_triggers.sql`)
5개 테이블에 대해 BEFORE UPDATE / BEFORE DELETE 트리거:
```sql
CREATE OR REPLACE FUNCTION prevent_update_or_delete() RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'WORM violation: % on % is forbidden', TG_OP, TG_TABLE_NAME
    USING ERRCODE = 'P0001';
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER worm_audit_log BEFORE UPDATE OR DELETE ON "AuditLog"
  FOR EACH ROW EXECUTE FUNCTION prevent_update_or_delete();
-- (반복: WormLog, AdminActionLog, DmsTriggerLog, ProofBundle)
```

## 레드팀 관점 (Phase C)
1. **WORM 우회** — `TRUNCATE` 가능한가? `ALTER TABLE DISABLE TRIGGER` 가능한가? Owner 권한으로 우회 가능한가?
2. **prevHash 단절** — genesis 블록 정의 우회·중복 genesis·해시 충돌 가능성.
3. **시크릿 leak** — DATABASE_URL의 password가 logs/.prisma cache/migration files에 남는가.
4. **SQL injection** — Prisma raw query 사용 시 위험 패턴 있는가.
5. **PII / Audit payload** — `payload Json` 필드에 평문 PII 저장 시 GDPR/암호화 가이드 누락.
6. **P5 체인 검증** — INSERT 시 prevHash 무결성 검증 트리거 부재 (현재 schema엔 INSERT 차단 트리거 없음 → 위반 INSERT 가능?). 검토 필요.
7. **Migration ordering** — `V_002_worm_triggers.sql`이 init보다 먼저 실행되면 테이블 부재로 실패.
8. **Postgres role 분리** — schema 적용용 admin vs 앱 runtime user 분리 부재. 앱 user가 직접 INSERT 가능.

심각도: BLOCKER/MAJOR/MINOR/SUGGESTION.
