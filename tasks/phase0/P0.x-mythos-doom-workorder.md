## 결정사항 (DOOM 5건)

**D1 — Runtime role 앱 경로 연결 (부분 처리)**
P0.x 단독 한계: Prisma multi-datasource 또는 별도 client 구성은 application layer. 이번엔:
- `infra/.env.example`에 `DATABASE_URL` (owner, migration용) vs `RUNTIME_DATABASE_URL` (runtime, 앱 INSERT/SELECT용) 두 변수 분리 + 주석.
- `packages/api/.env.example`에 두 URL 모두 명시 (owner=DATABASE_URL_OWNER, runtime=DATABASE_URL).
- 루트 `package.json` scripts: `prisma:migrate`는 owner URL 사용, app 실행(다음 Phase)은 runtime URL 사용 가이드 주석.
- **Phase 1 backlog 명시:** Prisma client가 두 datasource를 실제 사용하도록 application code 변경.

**D2 — Hash 재계산 검증 (부분 처리)**
P0.x 한계: canonical payload digest 정의는 application layer 영역. 이번엔:
- 새 migration `<TS>_hash_recompute_intent/migration.sql`: INSERT 트리거에 **CHECK 강화만** — `hash != prevHash` 보장(같은 값 차단), hash 길이 정확히 64자 lowercase hex 강제(이미 있음, 재확인).
- Trigger 함수 본문에 주석: `-- TODO[Phase 1+]: hash should be sha256(canonical(payload, prevHash, agentId/source/...))`
- **Phase 1 backlog 명시:** application에서 canonical payload encoding + sha256 재계산 후 INSERT.

**D3 — Payload PII 강화 (부분 처리)**
P0.x 한계: envelope encryption은 application layer. 이번엔:
- `prisma/schema.prisma` 5개 WORM 모델의 `payload Json` 옆 주석 강화 (현재 PII WARNING → 더 명확한 절차 표).
- WormLog `message String` 옆도 동일 경고 추가.
- **Phase 1 backlog 명시:** packages/api/src/audit/encrypt.ts 작성, INSERT 전 모든 payload digest+envelope 처리.

**D4 — Schema drift migration 추가 (완전 처리)** ★
새 migration `<TS>_drop_stale_audit_index/migration.sql`:
```sql
DROP INDEX IF EXISTS "AuditLog_hash_idx";
```
이걸로 schema와 DB가 정합. P0.x 단독 완료.

**D5 — Docker 부재 NOT_RUN (트랙 분리)**
P0.x 한계: Docker 호스트 환경. winget 자동 설치 실패. 이번엔:
- README "사전 요구사항"에 Docker Desktop 수동 설치 가이드 1줄 추가 + 대안 (WSL2 Docker Engine·Podman) 한 줄씩.
- **별도 트랙 명시:** Docker 설치는 사장님 결정 사항. 설치 후 별도 호출로 `docker compose config·up·psql·redis ping·down -v` 5게이트 재검증.
