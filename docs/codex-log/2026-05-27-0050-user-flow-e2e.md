# 실제 사용자 흐름 E2E 점검

## 무엇을 했나

실제 고객 개발자가 SDK를 붙이고, 보안 게이트를 호출하고, 감사자가 proof bundle을 조회하는 흐름을 기준으로 Phase 4~6 런타임을 다시 점검했다.

결론: 핵심 사용자 흐름은 E2E로 닫혔고, 점검 중 발견된 HumanGate 승인 아티팩트 정규화 결함을 수정했다.

## 왜

기존 테스트는 내부 서비스 단위 검증은 충분했지만, 실제 사용자가 겪는 순서인 `SDK -> Guard Verify -> receipt -> ProofBundle lookup`을 한 번에 통과시키는 검증이 부족했다.

또한 고위험 액션에서 HumanGate 승인 아티팩트를 함께 보내는 실제 요청은 일반 공개 context보다 깊고 긴 서명 페이로드를 포함하므로, HTTP ingress 정규화 계층과 컨트롤러 계층이 이를 안전하게 분리해서 처리하는지 확인할 필요가 있었다.

## 발견한 결함과 수정

1. signed HumanGate 승인 아티팩트가 TokenNorm max-depth 검사에서 일반 공개 context로 처리되어 요청이 차단됐다.
   - 수정: middleware에서 승인 아티팩트는 서명 검증용 원본으로 보존하고, 공개 context 검사는 유지했다.
2. 승인 아티팩트가 controller 이후 runtime payload에 남을 수 있었다.
   - 수정: verifier에서만 사용하고 RuntimeSeyer 입력에서는 제거했다.
3. E2E allowlist 환경값을 앱 생성 후 바꾸면 AllowListPolicyService 캐시와 어긋날 수 있었다.
   - 수정: 테스트가 실제 배포와 같게 앱 생성 전 env를 설정하도록 바꿨다.

## 리스크 / 미해결

- `COESITE_ALLOWED_ACTIONS`는 allowlist policy 캐시 때문에 프로세스 시작 전에 확정되어야 한다. 운영 문서와 배포 환경에서 이 순서를 지켜야 한다.
- `requestId`는 proof 조회 correlation key로 쓰이므로 PII를 넣지 않는 운영 규칙이 필요하다.
- PreviewBudget과 일부 in-memory/jsonl adapter는 MVP 런타임 검증용이다. 유료 운영 전에는 DB/S3/WORM adapter와 보관 정책 연결을 별도 운영 게이트로 닫아야 한다.
