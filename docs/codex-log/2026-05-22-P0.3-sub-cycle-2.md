# P0.3 sub-cycle 2 Phase B

## 무엇을 했나

P0.3 sub-cycle 2 Real WORM B1~B8을 구현했다.

- 기존 단일 `V_002_worm_triggers.sql`를 삭제하고 timestamped Prisma migration 3개로 분리했다.
- WORM 5개 테이블에 UPDATE/DELETE 차단, TRUNCATE 차단, chain INSERT 검증, hash/prevHash 제약을 추가했다.
- runtime DB role 분리 migration과 `RUNTIME_DATABASE_URL` 예시를 추가했다.
- WORM payload PII 경고, Agent `lastDecayAt`, `.pnpm-cache/` ignore를 반영했다.

## 왜

기존 WORM 구현은 UPDATE/DELETE 차단만 있었고, TRUNCATE 차단, prevHash 연결 검증, 중복 hash 방지, runtime 권한 분리가 없었다. 그 상태에서는 WORM이 문서상 선언에 가깝고 실제 DB 레벨 불변성·체인 무결성을 충분히 강제하지 못한다.

## 리스크 / 미해결

- Docker가 없어 migration 적용과 DB trigger/role 동작은 실제 Postgres에서 검증하지 못했다.
- `CREATE ROLE coesite_runtime`은 실제 DB 적용 시 migration 실행 계정에 role 생성 권한이 필요하다.
- `WormLog`는 기존 P0.3 스키마상 `payload` 필드가 없다. 새 payload 컬럼은 추가하지 않고 `message` 주변에 PII 금지 주석을 배치했다.
- 운영 runtime password는 placeholder 상태다. 실제 사용은 Phase 1+에서 KMS 또는 `*_FILE` secret 주입 방식 확정 후 진행해야 한다.
