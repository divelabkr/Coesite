# P0.3 sub-cycle 5 nano

## 무엇을 했나

P0.3 sub-5 nano 작업지시의 S1, S2를 `prisma/schema.prisma`에만 적용했다.

- S1: 4개 WORM 모델의 `prevHash` 단독 unique를 제거하고 scope별 복합 unique로 변경했다.
- S2: `AuditLog`의 `hash @unique`와 중복되는 `@@index([hash])`를 제거했다.

## 왜

기존 `prevHash` 단독 unique는 서로 다른 scope의 체인이 같은 이전 해시를 사용할 수 없게 만들어 schema drift를 만들 수 있었다.

`AuditLog.hash`는 이미 `@unique(map: "audit_hash_unique")`로 unique index를 만들기 때문에 별도 `@@index([hash])`가 중복이었다.

## 변경 파일

- `prisma/schema.prisma`
  - `WormLog.prevHash`의 단독 unique 제거.
  - `WormLog`에 `@@unique([source, prevHash], map: "worm_prev_unique")` 추가.
  - `AdminActionLog.prevHash`의 단독 unique 제거.
  - `AdminActionLog`에 `@@unique([adminId, prevHash], map: "admin_prev_unique")` 추가.
  - `DmsTriggerLog.prevHash`의 단독 unique 제거.
  - `DmsTriggerLog`에 `@@unique([triggerType, prevHash], map: "dms_prev_unique")` 추가.
  - `ProofBundle.prevHash`의 단독 unique 제거.
  - `ProofBundle`에 `@@unique([sessionId, prevHash], map: "proof_prev_unique")` 추가.
  - `AuditLog`의 `@@index([hash])` 제거.
- `docs/codex-log/2026-05-22-P0.3-sub-cycle-5-nano.md`
  - 이번 작업의 필수 요약본 작성.

## 리스크 / 미해결

- `packages/api/.env`가 없어 Prisma validate/format은 현재 PowerShell 세션 환경변수 주입이 필요했다. 실제 로컬 기본 게이트를 그대로 실행하려면 `.env.example` 기반의 `packages/api/.env` 준비가 필요하다.
- 이번 작업은 schema drift 수정만 수행했다. DB migration 생성 또는 적용은 작업지시 범위와 Docker 게이트 제외 조건에 따라 실행하지 않았다.
- 자체 codex review 호출은 하지 않았다.
