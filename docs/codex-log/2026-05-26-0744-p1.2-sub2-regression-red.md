# P1.2-sub2 Regression RED

## 무엇을 했나

결론: **회귀 테스트 8개 선행 단계 RED를 확인했다.**
이번 작업에서는 구현 파일을 수정하지 않고, P1.2-sub2 workorder의 보안 회귀 조건을 테스트로 먼저 고정했다.

Gemini CLI는 이번 배치에서 사용자 지시에 따라 waiver 처리했다. RedTeam Codex는 좁은 범위로 재가동했으나 제한 시간 안에 완료되지 않아 timeout/NOT_RUN으로 기록한다.

## 왜

현재 OraclePrevention은 header/size/time/exception/metadata 균일화가 완전하지 않다. 구현 전에 실패 테스트를 먼저 추가해야, 이후 수정이 실제로 취약점을 닫는지 확인할 수 있다.

## 검증 결과

### Unit RED

명령:

```bash
TMPDIR=/tmp pnpm exec vitest run packages/api/src/common/oracle-prevention/oracle-prevention.service.test.ts packages/api/src/common/oracle-prevention/uniform-error.filter.test.ts packages/api/src/meta-layer/token-norm/token-norm.middleware.test.ts
```

결과:

- Test Files: 3 failed
- Tests: 6 failed / 14 passed / 20 total

실패 요약:

- success EML이 아직 all-zero dummy다.
- 같은 rejection finding 100회 EML 값이 1개로 반복된다.
- oversized payload body가 10240 byte로 4096 cap을 넘는다.
- circular payload에서 `JSON.stringify` TypeError가 throw된다.
- UniformErrorFilter가 original exception metadata를 내부 기록하지 않는다.
- TokenNormMiddleware가 shared request clock 대신 자체 `performance.now()` 값을 넘긴다.

### E2E RED

명령:

```bash
TMPDIR=/tmp pnpm exec vitest run packages/api/test/e2e/app-module-runtime.e2e.test.ts
```

실행 조건:

- sandbox 내부는 `listen EPERM` 가능성이 있어 승인된 sandbox 밖 실행으로 확인했다.

결과:

- Test Files: 1 failed
- Tests: 3 failed / 7 passed / 10 total

실패 요약:

- AppModule success path에서도 EML이 all-zero dummy다.
- success response에서 `x-reason` header가 leak된다.
- thrown response에서 `x-reason` header가 leak된다.

## 리스크 / 미해결

- request clock storage는 현재 테스트에서 `coesiteRequestClockStartedAt` duck-typed property를 기대한다. Phase A에서 symbol/provider 방식으로 확정하면 이 테스트는 최종 설계에 맞게 정리해야 한다.
- internal metadata sink는 현재 console spy 기준으로 RED를 냈다. Phase A에서 injectable sink로 결정하면 테스트 기대값을 sink 기반으로 정리해야 한다.
- E2E는 sandbox 내부에서 안정 실행되지 않는다. sandbox 밖 승인 실행 결과를 분리 기록해야 한다.
