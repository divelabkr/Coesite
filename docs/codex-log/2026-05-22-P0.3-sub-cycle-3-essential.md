# P0.3 sub-3 Phase B essential

## 무엇을 했나

작업지시 C1~C5 중 이번 Phase B 범위만 적용했다. README는 수정하지 않았다.

- AuditLog 체인 검증을 agentId 기준으로 분리했다.
- agentId가 없는 WORM 계열 4개 테이블은 prevHash 전역 unique 구조를 유지했다.
- hash CHECK에서 GENESIS 허용을 제거하고, GENESIS는 prevHash 전용 sentinel로 남겼다.
- coesite_runtime role 생성에서 고정 placeholder password를 제거하고 `PASSWORD NULL` + `IF NOT EXISTS`로 바꿨다.
- 루트 Prisma 스크립트가 `packages/api/.env`를 dotenv-cli로 읽도록 바꿨고, dotenv-cli 의존성을 package.json 2곳에 선언했다.

## 왜

- 기존 `verify_chain_insert`는 AuditLog의 이전 해시 검증과 GENESIS 중복 검증을 테이블 전체 기준으로 수행해 agent별 체인 모델과 맞지 않았다.
- migration에 고정 password placeholder가 있어 운영 secret 주입 절차와 분리되지 않았다.
- role migration이 반복 실행에 취약했다.
- hash 컬럼이 GENESIS를 허용하면 실제 hash와 sentinel의 역할이 섞인다.
- Prisma 명령이 `packages/api/.env`를 읽지 않아 로컬 실행 시 DATABASE_URL 누락 가능성이 있었다.

## 변경 파일

- `prisma/migrations/20260522000002_chain_integrity/migration.sql`
  - 5개 WORM 테이블 hash CHECK를 hex hash 전용으로 변경.
  - AuditLog GENESIS 중복 검증과 prevHash 존재 검증을 `agentId` scope로 변경.
  - agentId 없는 4개 테이블의 `UNIQUE ("prevHash")` 구조 유지.
- `prisma/migrations/20260522000003_role_separation/migration.sql`
  - `CREATE ROLE ... PASSWORD 'CHANGEME...'` 제거.
  - `DO $$ ... IF NOT EXISTS ... CREATE ROLE coesite_runtime LOGIN PASSWORD NULL ... END $$;` 적용.
- `package.json`
  - `prisma:generate`, `prisma:migrate`, `prisma:reset`을 `dotenv -e packages/api/.env -- ...` 형태로 변경.
  - `dotenv-cli` devDependency 선언.
- `packages/api/package.json`
  - `dotenv-cli` devDependency 선언.
- `docs/codex-log/2026-05-22-P0.3-sub-cycle-3-essential.md`
  - Phase B 결과 기록.

## 리스크 / 미해결

- Docker가 없어 migration 적용과 trigger INSERT 동작은 실DB에서 검증하지 못했다.
- `pnpm-lock.yaml`은 영향 맵 밖이라 수정하지 않았다. 따라서 다음 install 단계에서 lockfile 갱신이 별도로 필요할 수 있다.
- `packages/api/.env` 파일이 현재 없어 bare Prisma validate는 실패한다. 이번 범위에서는 README나 env 샘플을 수정하지 않았다.
- C2의 runtime password 활성화 README 가이드는 사용자 보강 지시에 따라 Phase 1 backlog로 남겼다.
