# P1.1 TokenNorm

## 무엇을 했나
- `packages/api/src/meta-layer/token-norm/` 아래에 TokenNorm 서비스, NestJS 미들웨어, 모듈, 타입, export 파일을 신규 작성했다.
- Unicode NFKC, zero-width 제거, bidi 제어문자 제거, Cyrillic→Latin homoglyph 매핑, URL decode 1~2회 탐지, base64 decode 탐지를 구현했다.
- 변형 탐지 시 fail-closed로 동일한 `403` JSON 응답을 쓰고, 최소 처리 시간 padding을 적용했다.
- 감사 연동용으로 `req.tokenNormMetadata`에 탐지 경로, 원본, 정규화본, 사유, 시간 정보를 남기도록 했다.
- vitest 단위 테스트 29개를 추가했다.

## 왜
- P1.1 사양의 META-LAYER 첫 관문 요구에 따라 incoming request의 `body`, `query`, `headers`를 규칙 기반으로 정규화하고 인코딩 변형 우회를 차단해야 했다.
- 변형이 발견되면 정규화 후 통과가 아니라 거부해야 하므로, 미들웨어에서 fail-closed 응답을 직접 작성했다.
- 후속 감사 체인 연동을 위해 DB 쓰기 없이 메타데이터 구조만 남겼다.

## 리스크 / 미해결
- 사용자 지정 범위가 `packages/api/src/meta-layer/token-norm/` 디렉터리로 제한되어 `packages/api/src/app.module.ts`는 수정하지 않았다. 따라서 `TokenNormModule`은 준비됐지만 실제 앱 전역 적용은 별도 import 작업이 필요하다.
- base64 탐지는 printable payload를 변형으로 보고 거부한다. fail-closed 요구에 맞춘 동작이지만, 합법적인 base64 입력을 허용해야 하는 API가 생기면 allowlist 설계가 필요하다.
- class instance는 prototype mutation을 피하려고 순회하지 않는다. 일반 JSON request body/query/header는 plain object라 현재 범위에서는 문제가 없다.
- OraclePrevention 전역 인터셉터는 P1.1b 분리 범위라 구현하지 않았다.
