# P1.2-sub2 OraclePrevention 보안 재점검

## 무엇을 했나

결론: **현재 구현은 완벽하지 않으며, P1.2-sub2 구현 GO는 NO-GO**다.
`tasks/phase1/P1.2-sub2-essential-workorder.md`는 Phase A 이해 확인용 기준서로는 사용할 수 있지만, 현재 코드와 테스트가 그 기준을 아직 만족하지 않는다.

## 왜

현재 OraclePrevention 구현은 P8 외부 관찰 균일화의 일부만 구현되어 있다. 헤더 형식, size segment, 최소 지연은 일부 동작하지만 다음 취약점이 남아 있다.

- 성공 EML이 고정 더미값이라 반복 관찰로 성공/거부 경로 차이를 추론할 수 있다.
- 기존 handler/custom header를 제거하지 않고 새 header를 set만 한다.
- `headersSent === true`이면 로그만 남기고 return한다.
- padding/JSON 직렬화 실패가 uniform deny로 정규화되지 않고 throw 가능성이 있다.
- 4096 byte max cap, stable safe summary, circular/BigInt/large payload fail-closed 테스트가 아직 없다.
- request clock이 interceptor/filter/middleware 사이에서 공통화되지 않았다.
- exception original metadata sink가 아직 확정/구현되지 않았다.
- coverage 80% 게이트가 문서에는 있지만 `package.json`/CI에서 강제되지 않는다.

## 리스크 / 미해결

- P1.2-sub2는 아직 구현 전이다. 현재 코드에 남은 취약점은 workorder의 E1~E6 구현으로 닫아야 한다.
- 구현 전 RedTeam 재검토가 필요하다. 현재 이전 RedTeam은 장시간 미완료였고 workorder도 NOT_RUN이면 NO-GO로 둔다.
- coverage 80%는 별도 A-1 workorder로 먼저 강제하거나, 이번 구현 codex-log에서 명시 waiver를 받아야 한다.
- `headersSent` 경로는 workorder에 명시적으로 더 강하게 고정하는 것이 좋다. 이미 전송된 응답은 외부 uniform화가 불가능하므로 해당 경로를 내부 incident/fail-closed로 처리할 기준이 필요하다.
