# P0.3 sub-cycle 4 final

## 무엇을 했나

P0.3 sub-4 final Phase B의 M1, M2, M3를 작업지시 범위 안에서 적용했다.

- M1: `verify_chain_insert()`의 동적 SQL을 `TG_TABLE_SCHEMA`, `TG_TABLE_NAME` 조합으로 schema-qualified 처리했다.
- M2: `pnpm install`을 실행해 `dotenv-cli` 관련 lockfile 항목을 동기화했다.
- M3: 4개 WORM 테이블의 `prevHash` unique 제약과 insert trigger 검증을 scope-aware 기준으로 바꿨다.

## 왜

기존 trigger 동적 SQL은 table name만 사용해 같은 이름의 임시/다른 schema 테이블 공격면이 남아 있었다.

또한 `WormLog`, `AdminActionLog`, `DmsTriggerLog`, `ProofBundle`의 `prevHash`가 global unique라 multi-agent 또는 multi-scope chain 운용 시 정상 체인을 막을 수 있었다.

`dotenv-cli`는 `package.json`에 있었지만 lockfile 동기화가 필요했다.

## 변경 파일

- `prisma/migrations/20260522000002_chain_integrity/migration.sql`
  - 모든 trigger 동적 SQL을 `%I.%I` + `TG_TABLE_SCHEMA`, `TG_TABLE_NAME` 패턴으로 변경.
  - `WormLog`: `UNIQUE (source, "prevHash")`로 변경.
  - `AdminActionLog`: `UNIQUE ("adminId", "prevHash")`로 변경.
  - `DmsTriggerLog`: `UNIQUE ("triggerType", "prevHash")`로 변경.
  - `ProofBundle`: `UNIQUE ("sessionId", "prevHash")`로 변경.
  - GENESIS 중복 검증과 prior hash 검증을 각 테이블 scope별로 분기.
- `pnpm-lock.yaml`
  - `pnpm install` 결과로 `dotenv-cli` 및 관련 transitive dependency lock 항목 반영.
- `docs/codex-log/2026-05-22-P0.3-sub-cycle-4-final.md`
  - 이번 작업의 필수 요약본 작성.

## 리스크 / 미해결

- `prisma migrate dev`는 Docker CLI 부재로 실행하지 못했다. 실제 DB 적용 검증은 Docker 또는 PostgreSQL 접근 가능 환경에서 필요하다.
- 작업지시의 영향 맵 밖 파일 수정 금지에 따라 `prisma/schema.prisma`의 logical unique 선언은 의도적으로 바꾸지 않았다. 이후 Prisma schema와 DB migration 의도 간 정합성 확인이 필요할 수 있다.
- `prisma format`은 게이트 요구로 실행했다. 스키마 논리 변경은 하지 않았다.
